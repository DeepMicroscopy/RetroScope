\subsection{GPIO, I\textsuperscript{2}C, UART, and Power Wiring}\label{app:wiring}

The pin and address assignments are summarised in Table~\ref{tab:wiring}. The Sangaboard uses the Raspberry Pi UART on GPIO~14/15, which requires Bluetooth to be disabled so that the full UART is available. Additional Sangaboard control and step-signalling lines use GPIO~4, 8, 11, 23, 24, and 25. The rotary encoder is connected through GPIO~17 and 27 and passes through a 5\,V\ to 3.3\,V logic-level shifter to protect the Raspberry Pi inputs. The Z endstop uses GPIO~16, and the four user buttons use GPIO~6, 13, 19, and 26.

The I\textsuperscript{2}C bus on GPIO~2/3 is shared by the ADS1115 joystick ADC at address 0x48, two INA219 current-sense modules at 0x40 and 0x41, and the touchscreen controller/backlight at 0x38 and 0x45. All modules share common 5\,V, 3.3\,V, and ground rails. Figure~\ref{fig:fritzing} shows the corresponding Fritzing breadboard wiring overview. It includes the complete prototype wiring context, including the Raspberry Pi, motor controller, input devices, sensors, and power rails, rather than only the connectors on the custom PCB.

\subsection{Parametric Fusion Model Parameters}\label{app:fusion}

The parametric model is exported as \texttt{cad/ParametricModel.f3z}. The parameters in Table~\ref{tab:fusion} control the microscope-specific geometry: \texttt{connection\_hole} parameters set feet positions and radii, the \texttt{z\_assembly} parameters set the focus sub-assembly height and offsets, and the \texttt{xy\_assembly} offsets shift the XY sub-assembly. Adapting the platform to another microscope is intended to proceed by changing these parameters rather than rebuilding the assembly.

\subsection{Installation and Deployment Guide}\label{app:deploy}

The application is a Python/PySide6 project. A typical installation starts by cloning the repository, creating a virtual environment, and installing the Python requirements. On a desktop system the application can be started in mock mode with \texttt{python app.py --mock}. This allows UI, gallery, calibration, and workflow development without the physical microscope. The first launch may take a
little longer because Python creates its bytecode caches.

For Raspberry Pi deployment, the hardware interfaces must be prepared before the application is used: I\textsuperscript{2}C must be enabled for the ADS1115, INA219 modules, and touchscreen controller, and the full PL011 UART must be available for Sangaboard communication. In the prototype this means disabling Bluetooth so that GPIO~14/15 can be used as the stable UART interface. The physical wiring is described in Table~\ref{tab:wiring} and Figure~\ref{fig:fritzing}.

After the Python environment and hardware configuration are in place, the Raspberry Pi setup script is run once with sudo privileges using \texttt{sudo bash deploy/setup.sh}. The script links the repository into the expected application path if necessary, installs and enables the \texttt{retroscope.service} systemd service, adds the shutdown permission used by the UI, and applies the custom boot splash configuration. A reboot is required after this setup.

The service starts \texttt{deploy/start.sh \--\--scale 1.5}. This launcher activates an available virtual environment, uses a lock file to avoid multiple application instances, and restarts the application only when the application exits with the dedicated restart code. When running on the Raspberry Pi desktop without the service, the application can also be started manually under Wayland with the appropriate \texttt{WAYLAND\_DISPLAY} and \texttt{XDG\_RUNTIME\_DIR} environment variables. The running application provides the touchscreen UI locally and the REST API on port~8765 for remote capture access and selected actions.
