\subsection{Autofocus}

The autofocus workflow moves the Z axis through a configured search range and evaluates focus scores from the camera stream. The current position is the center of the search range, because the user usually begins near an approximate focus position. Then the workflow performs a coarse sweep up half the search range, then all the way down, followed by a fine sweep around the best coarse position. The workflow uses parabolic interpolation to refine the best position from the discrete samples. The autofocus service uses configured presets and parameters, including settle time, move-start delay, coarse and fine positions, samples per position, and minimum confidence. When minimum confidence is not met, the workflow returns to the start position.

The focus metric uses image sharpness cues derived from the luma/grayscale image. The implementation is guided by known autofocus approaches in microscopy \cite{pech2000diatom,sun2004autofocusing}, but it is adapted to the practical constraints of the camera path. The workflow validates that the camera analysis stream, Z motion, objective configuration, and UI state can work together. Figure~\ref{fig:autofocus-sweep} sketches the two-stage coarse/fine sweep and the parabolic peak refinement.

\subsection{Focus Stacking}

Focus stacking captures a sequence of images at different Z positions and combines them into a result with increased apparent depth of field. In RetroScope, focus stacking uses calibrated Z motion and objective-specific step settings. It captures native-resolution frames so that all planes have consistent dimensions and sufficient detail.

The final image uses OME-TIFF. It can store both the blended result and the individual frames as separate planes with coordinates. This is important because it preserves the raw acquisition data rather than only the processed stack result. External tools can inspect the original planes if needed.

The focus-stacking workflow is a useful validation case because it stresses several platform components at once: Z movement, blocking moves, camera capture, focus-related objective settings, metadata, and gallery display.

\subsection{Tile Scanning and Stitching}

Tile scanning captures a grid of adjacent fields of views and stores them either as a stitched panorama or as a grid mosaic. The tile planner uses image dimensions, objective pixel scale, overlap, and calibrated stage scales to compute motor-step spacing. If stage scale is wrong, tile spacing becomes visibly wrong. If backlash is not compensated, direction changes can create irregular overlap and make stitching less reliable.

The prototype supports different scan patterns: raster and serpentine. Raster became the default pattern because it avoids horizontal direction reversal on every row. Serpentine scanning is faster, but direction changes hurts the backlash compensation. The choice of raster scanning therefore reflects the mechanical limitations of the retrofit.

The implementation distinguishes between stitched panoramas and grid mosaics. A stitched panorama attempts image alignment and blending. A grid mosaic places full captured tiles into their grid cells while preserving overlap. The mosaic option is useful because it is deterministic and does not hide mechanical errors through image-registration behavior. The stitched option is closer to the expected user-facing result when the images contain enough texture and overlap for successful alignment. General tiled microscopy stitching is a well-established problem \cite{preibisch2009globally}.

\subsection{Gallery, Metadata, and Data Export}

The gallery workflow stores and displays snapshots, videos, focus stacks, and scans. It supports grid and list views, detail display, tags, metadata, thumbnails, and capture downloads through the REST API. Performance with a larger gallery was improved by generating thumbnails, instantiating only visible cells, and initializing detail content asynchronously.

Image metadata is part of the workflow. Captures include objective, position, dimensions, capture time, format, tags, and workflow-specific metadata. For images, OME-TIFF provides a metadata-rich storage format that can be opened in external microscopy tools. For external scripting, the REST API exposes capture summaries and download URLs, and the demo jupyter notebook shows a simple analysis workflow based on downloaded images.

\subsection{REST API for Remote Control and Capture Access}

RetroScope includes a FastAPI-based REST API that starts as part of the main application. It runs in a background Uvicorn thread while the Qt/QML interface remains the primary local user interface. The API is enabled by default on port 8765, provides Swagger UI at \texttt{/docs}, and exposes the OpenAPI document at \texttt{/openapi.json}. This allows a second computer on the same network to inspect the available endpoints, download captured data, or start selected actions without interacting with the touchscreen.

The current API intentionally exposes high-level application actions rather than low-level motor commands. Clients can check whether the API is alive, list gallery captures, download original capture files, trigger a still capture, and start autofocus. Capture listing supports filtering by type, sorting, limit, and offset. Downloads use an API-safe capture identifier derived from the stored capture path instead of exposing raw file paths directly.

Hardware-affecting actions are routed through the application service layer and marshalled back to the Qt thread, so remote requests use the same internal services as the local UI. The API is therefore a small remote-access surface for capture and workflow automation, not a separate motion-control
protocol. It can be extended with additional endpoints for more control and data access as needed.

\subsection{Role of the Workflows in Validating the Platform}

The workflows are not presented as independent research contributions. Autofocus, focus stacking, tile scanning, gallery management, and measurement are standards of digital microscopy functionality. Their role in this thesis is to validate the retrofit concept. Each workflow requires a different combination of mechanics, electronics, camera handling, calibration, and motion control.

If the platform can support these workflows with low-cost hardware and parametric mechanical adapters, then the core claim of the thesis is strengthened. Conversely, workflow failures identify where the retrofit approach reaches its limits. For example, tile scanning reveals stage-scale and backlash problems, autofocus reveals focus-score and frame-analysis timing problems, gallery performance reveals memory limitations on the Raspberry Pi, and video capture reveals bandwidth and pixel-format constraints.
